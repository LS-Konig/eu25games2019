## TL;DR

This repository contains a research-grade data cleaning and recoding pipeline for a 3-wave online survey on attitudes toward European integration, fielded in 2019 across 25 EU member states. The survey includes behavioral experiments (dictator/trust games, conjoint design) and a randomized priming experiment (Wave 2 only).

**Data Status:** Phases 0–7 complete (raw data → stacked waves). Phase 8 (value recoding) in progress. Preliminary cleaned data available at `data/03_final/eu25games2019.rds`, but **final coding scheme not yet finalized**.

**Note:** Intermediate files in `data/02_processed/` are **git-ignored** — the pipeline scripts (`code/*.qmd`) regenerate them. If you've cloned this repo, run the notebooks to generate intermediates before continuing Phase 8.

**Key Sample:** ~20,000 respondents × 3 waves × 25 countries

## Study Overview

### Survey Design

The **EU 25 Behavioral Games Survey** is a 3-wave online panel study (2019) examining how behavioral, experimental, and institutional factors shape attitudes toward European integration. The survey was fielded across 25 EU member states (Austria, Belgium, Bulgaria, Croatia, Czechia, Denmark, Estonia, Finland, France, Germany, Greece, Hungary, Ireland, Italy, Latvia, Lithuania, Netherlands, Poland, Portugal, Romania, Slovakia, Slovenia, Spain, Sweden, United Kingdom).

**Waves:**
- **Wave 1:** Demographics, political attitudes, baseline behavioral games
- **Wave 2:** Full behavioral suite (games + priming experiment), outcome batteries, party/vote items
- **Wave 3:** Stripped-down version of Wave 2 (priming removed), focused on outcome batteries and political behavior

### Experimental Components

#### Behavioral Games (Waves 2 & 3)
- **Dictator Game** (3 rounds): Allocating tokens between self and unknown partner
- **Trust Game** (3 rounds): Sequential exchange with asymmetric returns

Each respondent is randomly routed through 11 branching variants per game, with outcomes recorded for both players.

#### Conjoint Experiments
- **Wave 1:** Two-party choice experiment with randomized EU position and economic positions; respondent selects preferred party
- **Waves 2 & 3:** Candidate evaluation conjoint across 6 attributes (nationality, party, EU position, religion, age, gender)

#### Priming Experiments (All Waves)
Respondents randomly assigned to read information treatments about EU integration before completing outcome batteries. Treatment timing and engagement metrics recorded.
- **Wave 1:** Multiple priming conditions (details in questionnaire)
- **Wave 2:** Six conditions including valence reversals and issue-specific frames
- **Wave 3:** Streamlined priming protocol

## Variable Naming Scheme

All variables follow a semantic naming convention to distinguish raw responses, coded values, and experimental assignments.

### Prefixes

| Prefix | Category | Example | Notes |
|--------|----------|---------|-------|
| `q_` | Survey questions (coded) | `q_age`, `q_gender` | Clean numeric or factor; recoded values |
| `q_*_raw` | Survey questions (raw response) | `q_gender_raw` | Original string/value from survey; always kept |
| `cj_` | Conjoint experiment | `cj_dict1_pl1`, `cj_dict1_party_raw` | Dictator & trust game outcomes, attributes |
| `prime_` | Priming experiment | `prime_treatment`, `prime_firstclick` | Treatment assignment, timing data |
| `meta_` | Metadata/paradata | `meta_wave`, `meta_country`, `meta_pid` | Survey administration, respondent tracking |
| `ext_` | External merged data | `ext_party_id_pf_id` | Linked from external sources (PartyFacts, etc.) |
| `der_` | Derived variables | `der_co_partisan` | Computed from other variables |

### Special Cases & Suffixes

#### Language-Specific Variables
Some variables exist only in certain language versions of the survey due to country-specific options or translation gaps.

- **`_fra`:** Variables only in the French-language survey (e.g., French-specific party options, cultural references)
- **`_gre`:** Variables only in the Greek-language survey
- **[other language codes]:** Apply same convention for other country-specific variants (if present)

**Rule:** When a variable has a language-specific variant, it is **always paired** with a multilingual baseline variable:
- `q_party_id_raw` (multilingual) + `q_party_id_fra_raw` (French-only variant)

When coalescing, the baseline and language variant are merged using `coalesce()` so each respondent gets one value, prioritizing the language-specific response if available.

#### Branching & Duplication
- **`_b`:** Suffix for duplicate presentation due to survey branching (e.g., a question shown in two conditional branches appears as `Q7_1` and `Q7_1_b` before coalescing)

**Rule:** At most one non-`NA` value exists per respondent across all branches. These are coalesced before renaming.

#### Naming Examples (Coalescing in Action)

**Example 1 — Branching Duplication:**
```
Raw columns:    Q7_1 (branch A)  →  Coalesced to:  q_issue_marker_raw
                Q22_1 (branch B)   (each respondent has exactly one value)
```

**Example 2 — Numeric Suffix Normalization (Dynata export quirk):**
```
Raw columns:    Q4_29_TEXT (Austria)  →  Canonical:  Q4_TEXT  →  Renamed:  q_nationality_other_raw
                Q4_58_TEXT (Bulgaria)     (across countries)
```

**Example 3 — Outcome Battery (appears in multiple forms across waves):**
```
Wave 1: Q7_1–Q7_3 (EU efficacy items)  →  Wave 2: Q28_1–Q28_3 (same items)  →  Wave 3: Q35_1–Q35_3
All coalesced & renamed to:  q_eu_efficacy_understand_raw, q_eu_efficacy_care_raw, q_eu_efficacy_nosay_raw
                             (+ coded versions: q_eu_efficacy_understand, etc.)
```

### Finding Variables

See `data/02_processed/question_wording_map.csv` for the authoritative column-name → question-wording mapping (generated during Phase 0).

**A full codebook documenting each variable is forthcoming.** In the meantime:
- All demographic and outcome variables exist in both raw (`q_*_raw`) and coded (`q_*`) forms
- Behavioral game outcomes are prefixed `cj_` (conjoint/game)
- Priming treatment assignment and timing are prefixed `prime_`
- Metadata (wave, country, response ID, duration, etc.) are prefixed `meta_`

### 🔄 In Progress: Phase 8 — Value Recoding

**Objective:** Add clean coded versions of survey variables alongside raw responses.

**Current Approach (as implemented in `07_recode.qmd`):**

1. **Variable Signature Collection:** For each country subset, compute a "signature" (unique values for each column) to identify variables with identical response patterns across countries.

2. **Grouping:** Group columns by signature to find multilingual variants of the same question (e.g., gender coded as "Male"/"Female" in English vs. "Männlich"/"Weiblich" in German).

3. **Harmonization Strategy:**
   - **Categorical variables** (gender, religion, nationality, party): map all language variants to English canonical forms
   - **Numeric scales** (1–5 efficacy, 1–7 populism, etc.): verify scale properties, standardize "don't know" codes (8, 12) → `NA`
   - **Outcome batteries:** verify scale consistency across waves

4. **Implementation:** Add coded columns (`q_gender`, `q_religion`, etc.) to the stacked dataframe while preserving raw columns (`q_gender_raw`, `q_religion_raw`).

**Status:** 
- ✅ Gender variable harmonization implemented and tested
- ✅ Nationality variable mapping in progress

- ⏳ Final: save to `data/03_final/eu25games2019.rds`

## How to Contribute: Continuing Phase 8 Recoding

If you're a research assistant or collaborator joining this project to continue Phase 8 recoding:

### Quick Start

1. **Understand the current state:**
   ```r
   df_stacked <- readRDS("data/02_processed/waves_stacked.rds")
   
   # View column names to identify what's coded vs. raw
   names(df_stacked) |> grep("^q_", value = TRUE)
   
   # Check which raw variables need coding
   df_stacked |> 
     select(starts_with("q_")) |> 
     map_chr(class)
   ```

2. **Understand variable signatures** (see `07_recode.qmd` lines 92–122):
   - Each country subset is analyzed for unique response patterns
   - Variables with identical signatures are grouped (these are likely multilingual equivalents)
   - Use `unique()` and `table()` to inspect candidate variables

3. **Pick a variable to code:**
   - Prioritize demographic variables (gender, age, nationality, religion, class) — these are most stable across countries
   - Then political variables (party ID, vote choice, vote intention)
   - Finally outcome batteries (attitude scales, efficacy, populism items, etc.)
   - See `question_wording_map.csv` and survey questionnaires for what each raw variable represents

### Development Workflow

#### Step 1: Inspect Raw Values
```r
df_stacked |> 
  distinct(meta_country, q_age_raw) |>
  arrange(meta_country, q_age_raw) |>
  print(n = 100)
```

#### Step 2: Create a Mapping Table
```r
# Example: Age bands
age_map <- tibble(
  age_raw = c("18 - 24", "25 - 34", "35 - 44", "45 - 54", "55 - 64", "65+"),
  age_code = c(1, 2, 3, 4, 5, 6)
)
```

#### Step 3: Apply Recoding (Tidyverse Approach)
```r
df_stacked <- df_stacked |>
  mutate(
    q_age = case_match(
      q_age_raw,
      "18 - 24" ~ 1L,
      "25 - 34" ~ 2L,
      "35 - 44" ~ 3L,
      "45 - 54" ~ 4L,
      "55 - 64" ~ 5L,
      "65+" ~ 6L,
      .default = NA_integer_
    )
  )
```

#### Step 4: Validate
```r
# Check no data loss
df_stacked |> 
  mutate(lost = is.na(q_age) & !is.na(q_age_raw)) |>
  filter(lost) |>
  select(meta_country, q_age_raw)  # Should be empty

# Verify distribution
df_stacked |>
  count(q_age, meta_wave) |>
  pivot_wider(names_from = meta_wave, values_from = n)
```

#### Step 5: Document
Add a comment block in `07_recode.qmd` explaining:
- Which raw variable was coded
- Mapping logic (e.g., "Age bands to ordinal 1–6")
- Any edge cases or language-specific handling
- Data loss or unmapped values (if any)

### File Structure for Contributing

- **Edit in:** `code/07_recode.qmd` (add recoding code after the existing gender/nationality blocks)
- **Test with:** `data/02_processed/waves_stacked.rds` (read-only; create a local copy to test)
- **Output to:** `data/03_final/eu25games2019.rds` (final save; run at end of Phase 8)

### Conventions to Follow

1. **Always keep raw columns.** The coded column (`q_age`) goes alongside the raw (`q_age_raw`).
2. **Use `case_match()` not `recode().`** (Tidyverse modern standard; see `.claude/CLAUDE.md`)
3. **Recode "don't know" to `NA`.** Codes 8 and 12 typically mean "Don't know/No answer"; map these to `NA_integer_` or `NA_character_` as appropriate.
4. **Document language-specific variants.** If a variable has `_fra`, `_gre`, or other language suffixes, coalesce them (e.g., use `coalesce(q_party_fra, q_party_en)`) before final recoding.
5. **Test per wave.** Some variables may differ in scale or coding across waves; validate with `group_by(meta_wave)`.

### Questions & Troubleshooting

- **"What does this variable actually mean?"** → Check `data/02_processed/question_wording_map.csv` for the original question wording per country/wave.
- **"Are there language differences in a variable?"** → Search the raw column names for `_fra`, `_gre`, etc. suffixes, or inspect `unique(df_stacked$q_var_raw)` broken down by `meta_country`.
- **"How many respondents are missing for this variable?"** → `df_stacked |> count(is.na(q_var_raw))` tells you how many rows have missing raw data (can't be coded).
- **"Did I lose data in my recoding?"** → Run the validation check (Step 4 above); if any rows are `NA` for coded but non-`NA` for raw, fix the mapping.
